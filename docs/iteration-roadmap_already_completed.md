# 粮达健康 AI 智能营销迭代规划

日期：2026-06-15

## 1. 产品定位

粮达健康后续不定位为单纯的报告问答工具，也不定位为普通健康商城。

推荐定位：

```text
基于健康报告理解、Agent 记忆和多源健康画像的家庭健康智能营销系统。
```

核心能力：

```text
健康报告理解
+ 家庭健康画像
+ 长期互动记忆
+ 个性化餐单
+ 商品推荐转化
```

产品主线：

```text
上传报告
→ AI 抽取健康事实
→ 形成家庭健康画像
→ 结合手环近期状态
→ 结合互动记忆
→ Agent 生成餐单
→ 推荐商品
→ 展示推荐依据
→ 用户反馈写入记忆
```

## 2. 模块定位

| 模块 | 正确定位 | 不要承担的事 |
| --- | --- | --- |
| 个人档案 | 稳定健康主数据 | 不存对话偏好 |
| 报告解析 | 从 PDF/OCR 中抽取健康事实 | 不等同于 RAG |
| RAG 知识库 | 报告问答、原文证据追溯、多报告片段召回 | 不直接做餐单或商品推荐 |
| 健康事实库 | 承接报告抽取结果，形成长期健康证据 | 不做聊天上下文 |
| 手环数据 | 最近 7 天状态修正 | 不做长期健康结论 |
| 记忆系统 | 偏好、目标、营销反馈、跨会话个性化 | 不覆盖健康禁忌 |
| Agent | 意图识别、工具编排、自然语言输出 | 不把所有业务逻辑写进 prompt |
| 商城推荐 | 健康画像 + 记忆 + 商品标签转化 | 不做复杂电商系统 |

核心关系：

```text
报告抽取 → 健康事实
RAG → 查原文依据
手环 → 近期状态
记忆 → 个性化偏好
Agent → 编排
商城 → 营销转化
```

## 3. RAG 的边界

当前项目已有 PDF 上传、文本切片、embedding、Milvus 检索和成员隔离能力。

后续需要明确：RAG 不是健康事实抽取本身，也不是餐单和商品推荐的主决策层。

RAG 适合做：

- 报告问答，例如“爸爸报告里血脂怎么样”
- 原文依据追溯，例如“为什么说爸爸血脂偏高”
- 多报告片段召回，例如“找出近两年所有血脂相关片段”
- 给抽取任务提供候选片段，但抽取本身属于文档理解/信息抽取

RAG 不适合直接做：

- 餐单推荐
- 商品推荐
- 用户画像
- 营销决策
- 手环分析
- 长期记忆

正确分层：

```text
RAG = 查证据
健康事实 = 长期健康证据
健康画像 = 做决策
记忆 = 做个性化
Agent = 编排工具
```

## 4. 报告理解引擎

后续应把当前“RAG 知识库”升级为“健康报告理解引擎”。

报告上传后做两件事：

```text
1. 继续切 chunk + embedding，保留 RAG 证据追溯能力
2. 使用规则或 LLM 抽取健康事实，写入健康事实库
```

建议新增健康事实表：

```text
health_facts
```

建议字段：

```text
fact_id
member_id
fact_type: metric/risk/advice/recheck
name
value
unit
status: normal/warning/danger
source_document_id
source_page_no
source_chunk_id
evidence_text
created_at
```

示例：

```json
{
  "member_id": "mem_father",
  "fact_type": "risk",
  "name": "血脂偏高",
  "status": "warning",
  "source_document_id": "doc_2026_checkup",
  "source_page_no": 3,
  "evidence_text": "总胆固醇高于参考范围"
}
```

## 5. 健康画像聚合

当前项目已有 `HealthProfileService`，后续应升级为统一健康画像聚合器。

输入：

```text
个人档案
健康事实 health_facts
手环最近 7 天数据
互动记忆 memory
```

输出：

```text
长期风险：高血压、血脂偏高、骨密度低
近期状态：睡眠不足、步数偏低、血压近期偏高
饮食原则：低钠、少油、高钙、控糖
禁忌：过敏、健康禁忌
偏好：不爱吃鱼、偏好鸡肉、预算敏感
```

推荐决策优先级：

```text
过敏/禁忌
> 报告健康事实
> 个人健康标签
> 手环近期状态
> 当前问题
> 记忆偏好
> 营销转化
```

记忆不能覆盖健康安全约束。

示例：

```text
档案：爸爸高血压
记忆：爸爸喜欢咸口
```

系统不能推荐重盐食品，只能推荐低钠调味替代方案。

## 6. 记忆系统

记忆系统用于沉淀对话和行为中的偏好、阶段目标与营销反馈。

它不替代个人档案。

边界：

```text
个人档案管“这个人是谁、身体有什么硬约束”
记忆管“这个人最近怎么想、怎么选、怎么买”
```

选用 mem0 记忆框架。

记忆类型先保留四类：

```text
preference：喜欢/偏好
avoidance：不喜欢/排斥
goal：阶段目标
marketing_feedback：营销反馈
```

示例：

```text
爸爸不喜欢鱼
妈妈最近想控糖
用户经常购买杂粮主食
用户连续跳过高价商品
全家周末在家做饭
```


## 7. Agent 工具编排

后续不要只依赖一个大 prompt。Agent 应按意图调用明确工具。

建议工具：

```text
health_profile_tool
memory_search_tool
kb_search_tool
meal_plan_tool
mall_recommend_tool
cart_tool
```

工具职责：

| 工具 | 职责 |
| --- | --- |
| health_profile_tool | 获取成员或家庭健康画像 |
| memory_search_tool | 检索偏好、目标、营销反馈 |
| kb_search_tool | 只用于报告问答和依据追溯 |
| meal_plan_tool | 生成餐单 |
| mall_recommend_tool | 推荐商品 |
| cart_tool | 加入购物车 |

意图路由示例：

```text
报告问题 → kb_search
吃什么 → health_profile + memory_search + meal_plan
为什么推荐 → health_fact + kb_search 追溯依据
买什么 → health_profile + memory_search + mall_recommend
加入购物车 → cart_tool
```

## 8. 餐单到商品转化

这是 AI+智能营销的主 demo。

示例流程：

```text
用户：爸爸今晚吃什么？

Agent：
1. 读取爸爸健康画像
2. 读取爸爸相关记忆
3. 生成晚餐建议
4. 根据餐单和健康原则推荐商品
5. 给出推荐依据
```

示例输出：

```text
爸爸今晚建议低钠、少油、主食减量。
考虑到爸爸不喜欢鱼，可以用鸡胸肉或豆腐替代。

可选商品：
- 低钠酱油：适合血压偏高人群
- 杂粮饭：适合控脂和晚餐轻负担
- 无糖豆浆：补充蛋白质，避免高糖饮品
```

## 9. 推荐证据链

推荐证据链是重点。

商品详情或 Agent 回复中应展示：

```text
推荐给：爸爸
推荐原因：
- 健康事实：2026 体检报告提示血脂偏高
- 手环状态：最近 7 天步数偏低
- 互动记忆：爸爸不喜欢鱼
- 商品匹配：低脂、高纤、低钠
```

来源关系：

```text
健康事实 → 来自 health_facts
原文依据 → 通过 source_chunk_id 或 RAG/chunk 查证
手环状态 → 来自 device_daily_metrics
互动记忆 → 来自 Mem0
商品匹配 → 来自商城商品标签
```

## 10. 迭代优先级

### P0：统一定位和页面文案

目标：

- 将项目主线统一为“家庭健康智能营销 Agent”
- 避免页面和文档继续强化“报告问答工具”的单一印象

### P1：健康事实库

目标：

- 新增 `health_facts`
- 上传报告后抽取健康事实
- 健康事实保留文档、页码、chunk 和原文证据

### P2：健康画像接入健康事实

目标：

- `HealthProfileService` 读取 `health_facts`
- 报告抽取结果参与饮食原则、健康风险、商品推荐

### P3：记忆系统

目标：

- 新增 mem0 记忆框架
- 支持偏好、排斥、阶段目标、营销反馈
- Agent 推荐前读取相关记忆

### P4：Agent 意图识别和工具路由

目标：

- 区分报告问答、餐单推荐、商品推荐、依据追问、购物车动作
- `kb_search` 只用于报告问答和证据追溯

### P5：餐单接商品推荐

目标：

- `MealPlanService` 输出餐单后，提取健康原则和食材方向
- 复用商城商品标签和推荐规则
- Agent 回复中追加可选商品

### P6：推荐证据链展示

目标：

- Agent 回复和商品详情展示推荐依据
- 能追溯到健康事实、手环状态、互动记忆和商品标签

### P7：AI 技术增强

目标：

- 根据时间需要选择增强项
- 可选方向：
  - mem0 作为长期 Agent Memory 后端
  - Hybrid Search + rerank
  - 用户画像 embedding + 商品 embedding 语义推荐
  - LLM 推荐结果自检
  - 多 Agent 角色包装

## 11. AI 技术亮点

建议突出以下技术点：

```text
1. 健康报告理解引擎
   OCR + LLM 结构化抽取 + RAG 证据追溯

2. 多源健康画像
   档案 + 报告健康事实 + 手环近期状态

3. Agent Memory
   跨会话记录偏好、目标和营销反馈

4. Tool Calling Agent
   按意图调用报告检索、画像生成、餐单推荐、商品推荐工具

5. 语义化健康营销推荐
   从健康建议自然转化为商品推荐，并提供可解释依据
```

## 12. 推荐 demo 剧本

建议演示使用一个典型家庭场景。

准备数据：

```text
爸爸：高血压、BMI 偏高、报告提示血脂偏高
妈妈：骨密度偏低、希望补钙
孩子：成长发育期、偏好早餐类食品
```

演示流程：

```text
1. 创建家庭成员
2. 上传体检报告
3. AI 抽取健康事实
4. 同步最近 7 天手环数据
5. 用户告诉 Agent：爸爸不喜欢鱼
6. 系统写入互动记忆
7. 用户问：爸爸今晚吃什么
8. Agent 生成低钠、少油、主食减量晚餐
9. Agent 推荐低钠酱油、杂粮饭、无糖豆浆
10. 展示推荐证据链
11. 用户加入购物车，反馈写入营销记忆
```

这个 demo 能完整体现：

```text
健康报告理解
→ 多源画像
→ Agent 记忆
→ 个性化推荐
→ 营销转化
→ 反馈闭环
```
