# Prompt Engineering 工程架构设计

日期：2026-06-21

## 1. 目标

为 AI 竞赛方案产出一张**Prompt Engineering 工程架构图**，用于展示粮达健康在"LLM 调用链路的提示词工程"上的体系化设计。

定位：
- 与现有 `Context Engineering Layer`（`docs/context_engineering_arch.md`）平行
  - Context Engineering 负责"喂什么给模型"
  - Prompt Engineering 负责"怎么问模型"
- 聚焦**策略/技术维度**，而不是工程化生命周期维度
- 主体是 **Agent 循环**（Think → Act → Observe → Decide），不是单次 LLM 调用的横切图

不包含：
- Context Engineering 内部细节（已在另一张图说明）
- Prompt 模板版本控制、A/B 实验平台、CI/CD 等工程化流程（属于另一维度）
- 数据层、向量库、MySQL 等基础设施（属于总架构图）

## 2. 设计原则

1. **Agent 循环是心跳**
   所有能力模块（Prompt 装配、策略池、工具调用、输出解析、观测/评测/安全）都为 Agent 循环服务；不画成上下游流水线。

2. **与 Context Engineering 解耦**
   Prompt Assembly 的"Context 注入"槽位明确标注"来自 Context Engineering Layer"，避免两图职责重叠。

3. **策略池是可视化锚点**
   8 个推理策略（Zero-shot / Few-shot / CoT / ReAct / Self-Consistency / Reflection / ToT / Safety-Guard）独立画出来，体现技术深度；通过 Strategy Selector 动态编排。

4. **Safety 既是上下文也是策略**
   Safety-Guard 作为策略池的一员，把"安全约束"做成可调用的推理策略，而不是只挂在 Output 层兜底。

5. **可观测、可评测、可反馈**
   横切关注点层（Trace Logger / Eval Harness / Safety Watchdog / 反馈闭环）独立画出，体现"prompt 工程是工业化能力"。

```text
┌────────────────────────────────────────────────────────────┐
│              Prompt Engineering 架构                       │
│                                                            │
│   ┌──────────────────────────────────────────────────┐     │
│   │         Agent Loop (核心循环)                    │     │
│   │   Think → Act → Observe → Decide → (Re-Think)    │     │
│   └──────────────────────────────────────────────────┘     │
│           ▲             ▲              ▲          ▲        │
│           │             │              │          │        │
│   ┌───────┴──────┐ ┌────┴──────┐ ┌────┴─────┐ ┌───┴──────┐  │
│   │ ① Prompt     │ │ ② Strategy│ │ ③ Tool   │ │ ④ Output │  │
│   │  Assembly    │ │   Pool    │ │  Calling │ │  Parse   │  │
│   └──────────────┘ └───────────┘ └──────────┘ └──────────┘  │
│                                                            │
│  ──────────── ⑤ Observe / Eval / Safety (横切) ────────────  │
└────────────────────────────────────────────────────────────┘
```

**5 层能力云的职责切分**：
- ① Prompt Assembly：把多源内容装配成最终 prompt
- ② Strategy Pool：决定 Think 步用哪种推理策略
- ③ Tool Calling：Act 步的工具调用与结果归一化
- ④ Output Parsing & Validation：解析、校验、修复
- ⑤ Observe / Eval / Safety：横切关注点（Trace、评测、安全、反馈）

## 4. Agent 循环主体

```text
                       ┌──────────────────────────────┐
                       │   用户 Query + Member Scope  │
                       └──────────────┬───────────────┘
                                      ▼
                       ┌──────────────────────────────┐
                       │ ① Think：选策略 + 生成 plan  │◀──────┐
                       │   (从 Strategy Pool 选 ReAct │       │
                       │    / CoT / Reflection 等)    │       │
                       └──────────────┬───────────────┘       │
                                      ▼                       │
                       ┌──────────────────────────────┐       │
                       │ ② Act：调用工具 / 给出回答   │       │
                       │   (Tool: RAG / Memory /      │       │
                       │    Product / Calc...)        │       │
                       └──────────────┬───────────────┘       │
                                      ▼                       │
                       ┌──────────────────────────────┐       │
                       │ ③ Observe：解析工具结果      │       │
                       │   (Output Parsing + 校验)     │       │
                       └──────────────┬───────────────┘       │
                                      ▼                       │
                       ┌──────────────────────────────┐       │
                       │ ④ Decide：是否 Re-Think?     │       │
                       │   终止条件：                 │       │
                       │   - 拿到完整证据链           │       │
                       │   - Safety 触发硬终止        │       │
                       │   - 达到最大步数             │       │
                       └──────────────┬───────────────┘       │
                                      │                       │
                          ┌───────────┴──────────┐            │
                          ▼                      ▼            │
                      (终止：输出)        (继续：回 ① ) ──────┘
```

**为什么是四步而不是常见的三步**：
- 多出的 Decide 步是工程化的关键，它把"还要不要继续想"做成显式判定，便于接 Safety、步数限制、证据完整性等终止条件。
- 终结/再循环的分叉也明确画出来，避免被理解成"无限循环"。

**终止条件三件套**：
- 证据链完整：呼应证据链设计（`2026-06-17-evidence-chain-design.md`）
- Safety 触发硬终止：触发禁忌或敏感场景立即终止
- 最大步数：成本与延迟护栏

## 5. ① Prompt Assembly（提示词装配层）

```text
┌──────────────────────────────────────────────────────────┐
│                  Prompt Assembly                         │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │  System Prompt（角色 + 能力 + 边界）                │  │
│  │  - 身份：粮达健康家庭健康助手                       │  │
│  │  - 能力边界：解释报告 / 推荐餐单 / ...              │  │
│  │  - 不做什么：不诊断 / 不替代医生                    │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Context 注入（来自 Context Engineering Layer）    │  │
│  │  - member_context                                  │  │
│  │  - health_fact_context                             │  │
│  │  - memory_context (session + long_term)            │  │
│  │  - safety_context（硬约束置顶）                    │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Strategy Directive（当前步用啥策略）               │  │
│  │  - "请用 ReAct：先思考，再调用工具"                │  │
│  │  - "请用 CoT：分 3 步推理 ..."                     │  │
│  │  - "请用 Reflection：先评估上轮错误"               │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Few-shot Examples（按 intent 选）                 │  │
│  │  - 报告问答示例                                    │  │
│  │  - 餐单生成示例                                    │  │
│  │  - 商品推荐示例                                    │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Tool Schema（当前可用工具描述）                    │  │
│  │  - rag_search / memory_recall /                    │  │
│  │    product_search / rule_check ...                 │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Output Format Spec（约束输出 schema）              │  │
│  │  - JSON / 证据链字段 / 解释格式                    │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────┬───────────────────────────────────┘
                       ▼
            最终 Prompt Tokens（喂给 LLM）
```

**6 个槽位按"静态 → 动态"排序**：
1. System Prompt（最稳）
2. Context 注入（每次变）
3. Strategy Directive（每步可能变）
4. Few-shot Examples（按 intent 选）
5. Tool Schema（按 act 选）
6. Output Format Spec（按任务选）

**与 Context Engineering 的边界**：Context 槽位明确标注"来自 Context Engineering Layer"，装配层不重新生成上下文，只做"挑选 + 排序 + 注入"。

## 6. ② Strategy Pool（策略池）

```text
┌──────────────────────────────────────────────────────────┐
│                    Strategy Pool                          │
│              (Think 步骤按 intent 动态选)                 │
│                                                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐    │
│  │ Zero-shot    │  │ Few-shot     │  │ CoT          │    │
│  │ 直接回答     │  │ 样例驱动     │  │ 链式思考     │    │
│  │ 适用: 简单   │  │ 适用: 格式   │  │ 适用: 多步   │    │
│  │ 问答/查事实  │  │ 化输出       │  │ 推理         │    │
│  └──────────────┘  └──────────────┘  └──────────────┘    │
│                                                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐    │
│  │ ReAct        │  │ Self-        │  │ Reflection   │    │
│  │ 思考 + 行动  │  │ Consistency  │  │ 自我反思     │    │
│  │ 循环         │  │ 多路径投票   │  │ 适用:        │    │
│  │ 适用: 需调用 │  │ 适用: 事实性 │  │ Re-Think     │    │
│  │ 工具的复杂   │  │ 强、答案唯一 │  │ 步骤         │    │
│  │ 任务         │  │ 的判断       │  │              │    │
│  └──────────────┘  └──────────────┘  └──────────────┘    │
│                                                          │
│  ┌──────────────┐  ┌──────────────────────────────┐      │
│  │ ToT          │  │ Safety-Guard (硬约束 prompt) │      │
│  │ 思维树       │  │ - 触犯禁忌立即终止           │      │
│  │ 适用: 多方案 │  │ - 强制输出"安全替代"话术     │      │
│  │ 探索/对比    │  │                              │      │
│  └──────────────┘  └──────────────────────────────┘      │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │   Strategy Selector (选哪个)                        │  │
│  │   - 输入: intent + 上下文 + 上轮结果                 │  │
│  │   - 输出: 策略名 + 策略参数 (CoT 步数 / 示例数 ...)  │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**8 个策略的覆盖光谱**：
- 简单推理：Zero-shot / Few-shot
- 单路径推理：CoT
- 多路径推理：Self-Consistency
- 工具调用推理：ReAct
- 多方案探索：ToT
- 反思推理：Reflection
- 安全约束：Safety-Guard

**Strategy Selector 的输入决定策略**：
- `intent`：用户问什么 → 决定策略大类
- `上下文`：有没有硬约束 → 决定是否挂 Safety-Guard
- `上轮结果`：Re-Think 时是 Reflection 还是 Self-Consistency

**ReAct + Reflection 配对**：ReAct 应对循环的"第 1 次 Think"，Reflection 应对"Re-Think 步"，正好对应 Agent 循环的两类 Think 阶段。

## 7. ③ Tool Calling（工具调用层）

```text
┌──────────────────────────────────────────────────────────┐
│                  Tool Calling Layer                      │
│                                                          │
│  ┌──────────────────┐    ┌──────────────────────────┐    │
│  │  Tool Schema     │    │  Tool Registry           │    │
│  │  Generator       │───▶│  - 工具元数据             │    │
│  │  (从 Registry    │    │  - 参数 schema            │    │
│  │   生成 JSON      │    │  - 输入输出样例           │    │
│  │   Schema 注入    │    │  - 风险等级 (硬/软约束)   │    │
│  │   prompt)        │    │  - 耗时 / 成本预估        │    │
│  └──────────────────┘    └──────────────────────────┘    │
│           │                          │                   │
│           ▼                          ▼                   │
│  ┌──────────────────────────────────────────────────┐    │
│  │  Tool Router (路由：这一步用哪个/哪几个工具)     │    │
│  │  - 串行: 报告检索 → 画像聚合 → 推荐              │    │
│  │  - 并行: 同时调 RAG + Memory + Product            │    │
│  │  - 条件: 触发禁忌时改调 Safe-Alt-Generator       │    │
│  └──────────────────────────────────────────────────┘    │
│           │                                               │
│           ▼                                               │
│  ┌──────────────────┐    ┌──────────────────────────┐    │
│  │  Tool Executor   │───▶│  Tool Result Normalizer  │    │
│  │  (执行工具调用)  │    │  (统一格式：证据/状态/   │    │
│  │                  │    │   错误)                  │    │
│  └──────────────────┘    └──────────────────────────┘    │
│                                     │                    │
│                                     ▼                    │
│                        返回 Agent Loop 的 Observe 步      │
└──────────────────────────────────────────────────────────┘
```

**Tool Registry 是工程化的根**：
- 和 Strategy Pool 一样，工具是"可插拔能力"
- 元数据（参数 / 风险 / 耗时）让 Agent Loop 在 Think 步能"算"出怎么调

**Tool Router 3 种调用模式**：
- 串行：上一步结果影响下一步（如 RAG → 画像 → 推荐）
- 并行：相互独立的多源信息（RAG + Memory + Product）
- 条件分支：触犯禁忌时改调 Safe-Alt-Generator，对应 Safety Context 触发的关键链路

**Result Normalizer 作用**：把异构工具结果归一化成"证据 + 状态 + 错误"三字段，让 Observe 步好处理。

**风险等级字段**：让 Safety Context 能在 Tool 层就起作用，而不是等输出后兜底。

## 8. ④ Output Parsing & Validation

```text
┌──────────────────────────────────────────────────────────┐
│           ④ Output Parsing & Validation                  │
│                                                          │
│  ┌────────────────┐  ┌────────────────┐  ┌────────────┐  │
│  │ Format Parser  │  │ Schema         │  │ Evidence   │  │
│  │ (JSON / 结构化 │▶ │ Validator      │▶ │ Link       │  │
│  │  字段提取)     │  │ (必填字段/类型)│  │ Checker    │  │
│  └────────────────┘  └────────────────┘  └────────────┘  │
│           │                  │               │           │
│           └─────────┬────────┴───────┬───────┘           │
│                     ▼                ▼                   │
│            ┌──────────────┐  ┌──────────────────┐        │
│            │ 修复/重试     │  │ Safety Filter    │        │
│            │ (parse 失败  │  │ (输出后再次过     │        │
│            │  → 回 Agent  │  │  Safety Context) │        │
│            │  Loop 重做)  │  │                  │        │
│            └──────────────┘  └──────────────────┘        │
└──────────────────────────────────────────────────────────┘
```

**Output 解析三层递进**：
1. Format Parser：把 LLM 输出解析为结构化字段
2. Schema Validator：检查必填字段、类型、范围
3. Evidence Link Checker：检查证据链是否完整（呼应证据链设计）

**每层失败的兜底动作不同**：
- 解析失败：回 Agent Loop 重做（Think 步改用 Few-shot + Output Format Spec）
- Schema 不通过：修复 / 重试
- 证据链不完整：补 RAG / 记忆召回
- Safety Filter 触发：直接拒绝或替换

**Evidence Link Checker 是项目亮点**：把"为什么这么推荐"做成机器可校验的步骤，而不是只展示给用户。

## 9. ⑤ Observe / Eval / Safety（横切关注点）

```text
┌──────────────────────────────────────────────────────────┐
│            ⑤ Observe / Eval / Safety                     │
│                  (横切关注点)                            │
│                                                          │
│  ┌────────────────┐  ┌────────────────┐  ┌────────────┐  │
│  │ Prompt Trace   │  │ Eval Harness   │  │ Safety     │  │
│  │ Logger         │  │ (评测 Harness) │  │ Watchdog   │  │
│  │ - 每次调用的   │  │ - 固定用例集   │  │ - 实时检测 │  │
│  │   prompt + 输出│  │ - 引用证据?    │  │   触犯禁忌 │  │
│  │ - 证据链快照   │  │ - 成员对吗?    │  │   的中间   │  │
│  │ - Context 用/   │  │ - 推荐合规?    │  │   步骤     │  │
│  │   弃记录       │  │ - 输出可解释?  │  │ - 立即终止 │  │
│  └────────────────┘  └────────────────┘  │   并回退   │  │
│                                          └────────────┘  │
│           │                   │                │         │
│           └─────────┬─────────┴────────┬───────┘         │
│                     ▼                  ▼                 │
│            ┌──────────────────┐  ┌──────────────────┐    │
│            │ Cost / Latency   │  │ 反馈闭环         │    │
│            │ Monitor          │  │ (用户行为 →      │    │
│            │ (token / 步数 /  │  │  记忆 → 下次     │    │
│            │  工具耗时)       │  │  Selector 输入)  │    │
│            └──────────────────┘  └──────────────────┘    │
└──────────────────────────────────────────────────────────┘
```

**横切 4 件套**：

1. **Prompt Trace Logger**
   - 记录每次 LLM 调用的 prompt、输出、证据链快照
   - 与 Context Snapshot 配合：记录用了/丢了哪些 context
   - 用途：调试、回放、竞赛展示、Context Eval

2. **Eval Harness**
   - 来自项目已有的"评测 Harness"（`ai-competition-technical-solution.md` 第 7 段）
   - 固定用例集：报告问答 / 成员识别 / 证据引用 / 健康禁忌 / 餐单生成 / 商品推荐 / 反馈记忆
   - 用途：验证 prompt 改动不退化

3. **Safety Watchdog**
   - 过程级安全检查，不只查最终输出
   - 实时检测中间步骤是否触犯禁忌
   - 触发立即终止并回退到 Safe-Alt-Generator

4. **反馈闭环**
   - 用户行为（点击/跳过/收藏/价格敏感度/口味反馈）→ 记忆沉淀
   - 记忆进入 Strategy Selector 的输入
   - 形成数据飞轮：用户反馈 → 策略优化 → 更好推荐

**Cost / Latency Monitor 独立**：token 数 / 步数 / 工具耗时是 prompt 工程的成本护栏，不只是"埋点"。

## 10. 与现有架构的衔接

| 现有模块 | 在 Prompt Engineering 架构中的位置 |
| --- | --- |
| Context Engineering Layer（5 类 Context） | Prompt Assembly 的"Context 注入"槽位 |
| Skill 拆分 | Tool Calling 的 Tool Registry 元数据 |
| 证据链设计 | ④ Evidence Link Checker + ⑤ Prompt Trace Logger |
| 评测 Harness | ⑤ Eval Harness |
| 长期记忆（mem0） | 反馈闭环 → Strategy Selector 输入 |
| Safety Context 优先级 | Safety-Guard 策略 + Safety Watchdog + Safety Filter |

## 11. 后续可选扩展（不在本期范围）

- Prompt 模板版本控制 + A/B 实验平台（属于工程化生命周期维度）
- Prompt 自动调优（AutoPrompt / OPRO）
- 多模型路由（不同 LLM 不同 prompt 模板）
- 流式输出 + 增量解析
- Prompt 注入攻击检测

## 12. 验收口径

- 一张架构图能清晰展示：Agent 循环 + 5 层能力云 + 8 个策略 + 横切 4 件套
- 与 Context Engineering 架构图职责不重叠、引用关系明确
- 竞赛评委能在 1 分钟内看出"这是工业化 prompt 工程，而不是调 prompt"
