# 粮达健康 AI 智能营销迭代规划

日期：2026-06-15

## 1. 产品定位

粮达健康后续不定位为单纯的报告问答工具，也不定位为普通健康商城。

推荐定位：

```text
基于健康报告理解、Agent 记忆和多源健康画像的家庭健康智能营销系统。
```

核心能力：

```text
健康报告理解
+ 家庭健康画像
+ 长期互动记忆
+ 个性化餐单
+ 商品推荐转化
```

产品主线：

```text
上传报告
→ AI 抽取健康事实
→ 形成家庭健康画像
→ 结合手环近期状态
→ 结合互动记忆
→ Agent 生成餐单
→ 推荐商品
→ 展示推荐依据
→ 用户反馈写入记忆
```

## 2. 模块定位

| 模块 | 正确定位 | 不要承担的事 |
| --- | --- | --- |
| 个人档案 | 稳定健康主数据 | 不存对话偏好 |
| 报告解析 | 从 PDF/OCR 中抽取健康事实 | 不等同于 RAG |
| RAG 知识库 | 报告问答、原文证据追溯、多报告片段召回 | 不直接做餐单或商品推荐 |
| 健康事实库 | 承接报告抽取结果，形成长期健康证据 | 不做聊天上下文 |
| 手环数据 | 最近 7 天状态修正 | 不做长期健康结论 |
| 记忆系统 | 偏好、目标、营销反馈、跨会话个性化 | 不覆盖健康禁忌 |
| Agent | 意图识别、工具编排、自然语言输出 | 不把所有业务逻辑写进 prompt |
| 商城推荐 | 健康画像 + 记忆 + 商品标签转化 | 不做复杂电商系统 |

核心关系：

```text
报告抽取 → 健康事实
RAG → 查原文依据
手环 → 近期状态
记忆 → 个性化偏好
Agent → 编排
商城 → 营销转化
```

## 3. RAG 的边界

当前项目已有 PDF 上传、文本切片、embedding、Milvus 检索和成员隔离能力。

后续需要明确：RAG 不是健康事实抽取本身，也不是餐单和商品推荐的主决策层。

RAG 适合做：

- 报告问答，例如“爸爸报告里血脂怎么样”
- 原文依据追溯，例如“为什么说爸爸血脂偏高”
- 多报告片段召回，例如“找出近两年所有血脂相关片段”
- 给抽取任务提供候选片段，但抽取本身属于文档理解/信息抽取

RAG 不适合直接做：

- 餐单推荐
- 商品推荐
- 用户画像
- 营销决策
- 手环分析
- 长期记忆

正确分层：

```text
RAG = 查证据
健康事实 = 长期健康证据
健康画像 = 做决策
记忆 = 做个性化
Agent = 编排工具
```

## 4. 报告理解引擎

后续应把当前“RAG 知识库”升级为“健康报告理解引擎”。

报告上传后做两件事：

```text
1. 继续切 chunk + embedding，保留 RAG 证据追溯能力
2. 使用规则或 LLM 抽取健康事实，写入健康事实库
```

建议新增健康事实表：

```text
health_facts
```

建议字段：

```text
fact_id
member_id
fact_type: metric/risk/advice/recheck
name
value
unit
status: normal/warning/danger
source_document_id
source_page_no
source_chunk_id
evidence_text
created_at
```

示例：

```json
{
  "member_id": "mem_father",
  "fact_type": "risk",
  "name": "血脂偏高",
  "status": "warning",
  "source_document_id": "doc_2026_checkup",
  "source_page_no": 3,
  "evidence_text": "总胆固醇高于参考范围"
}
```

## 5. 健康画像聚合

当前项目已有 `HealthProfileService`，后续应升级为统一健康画像聚合器。

输入：

```text
个人档案
健康事实 health_facts
手环最近 7 天数据
互动记忆 memory
```

输出：

```text
长期风险：高血压、血脂偏高、骨密度低
近期状态：睡眠不足、步数偏低、血压近期偏高
饮食原则：低钠、少油、高钙、控糖
禁忌：过敏、健康禁忌
偏好：不爱吃鱼、偏好鸡肉、预算敏感
```

推荐决策优先级：

```text
过敏/禁忌
> 报告健康事实
> 个人健康标签
> 手环近期状态
> 当前问题
> 记忆偏好
> 营销转化
```

记忆不能覆盖健康安全约束。

示例：

```text
档案：爸爸高血压
记忆：爸爸喜欢咸口
```

系统不能推荐重盐食品，只能推荐低钠调味替代方案。

## 6. 记忆系统

记忆系统用于沉淀对话和行为中的偏好、阶段目标与营销反馈。

它不替代个人档案。

边界：

```text
个人档案管“这个人是谁、身体有什么硬约束”
记忆管“这个人最近怎么想、怎么选、怎么买”
```

选用 mem0 记忆框架。

记忆类型先保留四类：

```text
preference：喜欢/偏好
avoidance：不喜欢/排斥
goal：阶段目标
marketing_feedback：营销反馈
```

示例：

```text
爸爸不喜欢鱼
妈妈最近想控糖
用户经常购买杂粮主食
用户连续跳过高价商品
全家周末在家做饭
```


## 7. Agent 工具编排

后续不要只依赖一个大 prompt。Agent 应按意图调用明确工具。

建议工具：

```text
health_profile_tool
memory_search_tool
kb_search_tool
meal_plan_tool
mall_recommend_tool
cart_tool
```

工具职责：

| 工具 | 职责 |
| --- | --- |
| health_profile_tool | 获取成员或家庭健康画像 |
| memory_search_tool | 检索偏好、目标、营销反馈 |
| kb_search_tool | 只用于报告问答和依据追溯 |
| meal_plan_tool | 生成餐单 |
| mall_recommend_tool | 推荐商品 |
| cart_tool | 加入购物车 |

意图路由示例：

```text
报告问题 → kb_search
吃什么 → health_profile + memory_search + meal_plan
为什么推荐 → health_fact + kb_search 追溯依据
买什么 → health_profile + memory_search + mall_recommend
加入购物车 → cart_tool
```

## 8. 餐单到商品转化

这是 AI+智能营销比赛的主 demo。

示例流程：

```text
用户：爸爸今晚吃什么？

Agent：
1. 读取爸爸健康画像
2. 读取爸爸相关记忆
3. 生成晚餐建议
4. 根据餐单和健康原则推荐商品
5. 给出推荐依据
```

示例输出：

```text
爸爸今晚建议低钠、少油、主食减量。
考虑到爸爸不喜欢鱼，可以用鸡胸肉或豆腐替代。

可选商品：
- 低钠酱油：适合血压偏高人群
- 杂粮饭：适合控脂和晚餐轻负担
- 无糖豆浆：补充蛋白质，避免高糖饮品
```

## 9. 推荐证据链

推荐证据链是参赛展示重点。

商品详情或 Agent 回复中应展示：

```text
推荐给：爸爸
推荐原因：
- 健康事实：2026 体检报告提示血脂偏高
- 手环状态：最近 7 天步数偏低
- 互动记忆：爸爸不喜欢鱼
- 商品匹配：低脂、高纤、低钠
```

来源关系：

```text
健康事实 → 来自 health_facts
原文依据 → 通过 source_chunk_id 或 RAG/chunk 查证
手环状态 → 来自 device_daily_metrics
互动记忆 → 来自 Mem0
商品匹配 → 来自商城商品标签
```

## 10. 迭代优先级

### P0：统一定位和页面文案

目标：

- 将项目主线统一为“家庭健康智能营销 Agent”
- 避免页面和文档继续强化“报告问答工具”的单一印象

### P1：健康事实库

目标：

- 新增 `health_facts`
- 上传报告后抽取健康事实
- 健康事实保留文档、页码、chunk 和原文证据

### P2：健康画像接入健康事实

目标：

- `HealthProfileService` 读取 `health_facts`
- 报告抽取结果参与饮食原则、健康风险、商品推荐

### P3：记忆系统

目标：

- 新增 mem0 记忆框架
- 支持偏好、排斥、阶段目标、营销反馈
- Agent 推荐前读取相关记忆

### P4：Agent 意图识别和工具路由

目标：

- 区分报告问答、餐单推荐、商品推荐、依据追问、购物车动作
- `kb_search` 只用于报告问答和证据追溯

### P5：餐单接商品推荐

目标：

- `MealPlanService` 输出餐单后，提取健康原则和食材方向
- 复用商城商品标签和推荐规则
- Agent 回复中追加可选商品

### P6：推荐证据链展示

目标：

- Agent 回复和商品详情展示推荐依据
- 能追溯到健康事实、手环状态、互动记忆和商品标签

### P7：AI 技术增强

目标：

- 根据时间和比赛需要选择增强项
- 可选方向：
  - mem0 作为长期 Agent Memory 后端
  - Hybrid Search + rerank
  - 用户画像 embedding + 商品 embedding 语义推荐
  - LLM 推荐结果自检
  - 多 Agent 角色包装

## 11. AI 技术亮点包装

比赛中建议突出以下技术点：

```text
1. 健康报告理解引擎
   OCR + LLM 结构化抽取 + RAG 证据追溯

2. 多源健康画像
   档案 + 报告健康事实 + 手环近期状态

3. Agent Memory
   跨会话记录偏好、目标和营销反馈

4. Tool Calling Agent
   按意图调用报告检索、画像生成、餐单推荐、商品推荐工具

5. 语义化健康营销推荐
   从健康建议自然转化为商品推荐，并提供可解释依据
```

## 12. 推荐 demo 剧本

建议参赛演示使用一个典型家庭场景。

准备数据：

```text
爸爸：高血压、BMI 偏高、报告提示血脂偏高
妈妈：骨密度偏低、希望补钙
孩子：成长发育期、偏好早餐类食品
```

演示流程：

```text
1. 创建家庭成员
2. 上传体检报告
3. AI 抽取健康事实
4. 同步最近 7 天手环数据
5. 用户告诉 Agent：爸爸不喜欢鱼
6. 系统写入互动记忆
7. 用户问：爸爸今晚吃什么
8. Agent 生成低钠、少油、主食减量晚餐
9. Agent 推荐低钠酱油、杂粮饭、无糖豆浆
10. 展示推荐证据链
11. 用户加入购物车，反馈写入营销记忆
```

这个 demo 能完整体现：

```text
健康报告理解
→ 多源画像
→ Agent 记忆
→ 个性化推荐
→ 营销转化
→ 反馈闭环
```

## 13. 长期商用规划（P8-P12，3-5 年集团级平台）

日期：2026-06-22

### 13.1 一句话定位

> **粮达健康 = 集团级"家庭健康智能服务"平台**：C 端是高频"今晚吃什么"决策入口；B 端是给中粮各品牌用的"家庭健康营销云"；两者通过"家庭健康图谱"和"全集团统一货架"双轮驱动。

### 13.2 核心飞轮

```text
  C 端家庭用户沉淀家庭健康事实
            ↓
  家庭健康图谱越来越精准
            ↓
  集团品牌 → 智能匹配 + 智能营销
            ↓
  用户收到更对路、更合身的"吃什么 / 买什么"
            ↓
  复购、订阅、推荐数据反哺图谱
```

### 13.3 五阶段路线图

| 阶段 | 时间 | 主题 | 关键交付 | 商业化 |
|---|---|---|---|---|
| **P8 比赛 + 集团 PoC** | 0-6 个月 | 把"集团级平台"讲清楚、做出 PoC | 比赛提交物 + 集团领导版 Demo + 1-2 个品牌跑通链路 | 无（投入期） |
| **P9 C 端 MVP + B 端 SaaS 起步** | 6-18 个月 | C 端公测上线 + B 端给品牌部出 SaaS | 全集团统一货架 MVP（小程序 / App）；品牌 SaaS 1.0（家庭健康画像引擎 + AI 营销助手） | B 端 SaaS 收费（年付 / 项目制）；C 端小范围会员订阅内测 |
| **P10 多品牌 + 多设备** | 18-36 个月 | 全集团品牌接入 + IoT 入口成型 | 智能硬件 SDK/对接（手环 / 血压计 / 秤 / 智能冰箱食谱识别）；中粮 5+ 品牌生产级接入；C 端日活 50 万级 | C 端会员订阅正价上线；B 端 SaaS 多品牌订阅；数据洞察报告 |
| **P11 双边飞轮 + 金融合作** | 36-48 个月 | C 端 B 端数据互通起飞 + 金融服务试点 | AI 主厨 / AI 营养师（订阅升级）；与中粮资本 / 险企合作的健康险、营养险；行业级"食品-健康"知识图谱 | 金融合作分成；高级订阅；B 端数据服务 |
| **P12 集团基础设施化** | 48-60 个月 | 成为集团数字基建、可对外输出 | 标准化产品 / API / SDK；为外部品牌、地方政府、保险、医院等输出 | 多边平台收入；品牌外部合作；行业数据 SaaS |

### 13.4 家庭健康图谱（数据飞轮的核心）

定位：粮达健康最值钱的资产，是一张持续增长、不可被竞争对手复制的"家庭 × 健康 × 食品"知识图谱。

```text
L1 家庭健康事实（长期）
  - 报告指标、慢性风险、过敏、年龄、性别、身高体重
  - 来源：体检报告 / 问卷 / 家庭档案
  - 用法：个性化主决策层

L2 家庭健康状态（近 7-30 天）
  - 手环步数、睡眠、血压、心率、近期饮食、餐厨行为
  - 来源：IoT 设备 + 主动输入
  - 用法：状态修正层

L3 家庭消费偏好（滚动 12 个月）
  - 购买记录、品牌偏好、价格敏感、复购周期、互动反馈
  - 来源：平台交易 + Agent 记忆
  - 用法：营销转化层
```

设计要点：
- 三层不混在一张表，按"长期事实 / 近期状态 / 滚动偏好"分别存储
- L1 是 LLM 抽取的结构化事实，沿用现有 `health_facts` 扩展
- L2 是设备时序数据，P10 起引入时序库
- L3 是事件流，沉淀到 Agent 记忆与营销标签库
- 三层都在"家庭"粒度下聚合（关键差异化点：别家都在"个人"粒度）

### 13.5 集团对接与品牌接入

集团治理三层：

```text
       集团总部 / CIO
            ↓ 战略 + 预算 + 数据治理
       粮达健康（平台）
       ↗          ↖
  福临门   蒙牛   中茶   长城   家佳康 ...
  （品牌接入方）
```

品牌接入路径（每批 1-2 个，跑通再扩）：

| 步骤 | 内容 | 周期 | 责任方 |
|---|---|---|---|
| B1 数据标准化 | 品牌商品标签、营养成分、库存、价签、活动 API 标准化 | 4-6 周 | 品牌方 IT + 平台 |
| B2 智能营销接入 | 把家庭健康画像引擎接入品牌投放系统 / 营销云 | 4-8 周 | 平台 + 品牌市场部 |
| B3 联合产品上线 | 在粮达健康内做"品牌专区"或"主题活动" | 2-4 周 | 平台运营 + 品牌运营 |
| B4 复盘与扩量 | 看 GMV / 复购 / 留存，跑通后扩展 SKU / 活动类型 | 持续 | 双边 |

对接协议（避免做两层平行实现）：

```text
数据回流：品牌商品 × 用户偏好 = 标签匹配服务（API 形式）
营销工具：AI 营销助手作为品牌部 SaaS（不是品牌部另起一套）
商品库存：复用品牌现有 ERP / 中粮"我买网"基础设施（如果还在跑）
```

### 13.6 商业化矩阵

| 收入线 | 阶段首发 | 客单价量级 | 规模 / 利润特征 |
|---|---|---|---|
| C 端会员订阅 | P9 内测 / P10 正价 | ¥99-¥399/年 | 高频小额、规模化、留存敏感 |
| C 端增值服务（AI 主厨 / 营养师 / 私人餐单） | P11 | ¥9.9-¥99/月 | 高毛利、低基数、高增长 |
| B 端 SaaS（品牌部用画像引擎 + AI 营销助手） | P9 收项目费 / P10 转年订阅 | ¥50 万-¥500 万/年/品牌 | 大额、稳定、政企关系 |
| 数据洞察报告（行业 / 趋势 / 消费者） | P10 | ¥10 万-¥100 万/份 | 高毛利、品牌方市场部买单 |
| 金融合作分成（健康险、营养险、消费分期） | P11 | 视合作 | 增量分成 |
| 广告 / 品牌曝光 | P10 谨慎开启 | CPM / CPC | 流量规模化后才有意义，要克制 |

明确不做（避免过度设计）：
- **不做** 电商订单履约、客服、退货（走中粮"我买网"/ 品牌方履约）
- **不做** 自营仓配（仅做"商品 + 内容 + 推荐"）
- **不做** 多类目横向扩张（守住"吃 + 健康吃"）
- **不做** 医疗问诊 / 在线问药（合规硬约束）

### 13.7 技术架构升级

底座：沿用现有 FastAPI + Vue/TS + Vite + Milvus + Mem0 + LangChain Agent；增量升级，不另起炉灶。

| 维度 | 现状 | P10-P12 增量 |
|---|---|---|
| 前端 | 单端 Web | App（iOS / Android）+ 小程序 + Web + 智能音箱 / 智能冰箱对话面板 |
| AI | LangChain 单 Agent | 多 Agent 协同（健康管家 / 餐厨助手 / 营养师 / 品牌助手）；多模态（图谱 / 语音 / OCR） |
| 记忆 | Mem0 单层 | 三层记忆：长期事实（结构化）+ 短期上下文（窗口）+ 营销反馈（事件流） |
| 检索 | Milvus 单一向量 | Hybrid Search（向量 + BM25 + KG）+ rerank；引入"家庭健康图谱"专用索引 |
| 数据 | MySQL + Milvus + 文件 | 加时序库（IoT 设备数据）+ 特征库（家庭健康画像）+ 标签图谱（Neo4j 或类似） |
| 合规 | 无 | 数据脱敏 / 加密存储 / 差分隐私 / 合规审计 / 算法备案 |
| 多端协议 | 无 | 标准化 OpenAPI + SDK + 智能硬件 SDK（IoT 厂商对接） |

增量原则（避免过度设计）：
- 不为多端单独搭一套后端；后端只暴露稳定 API，前端多端共享后端
- 不为多模态单独搭一套推理；多模态能力作为 Agent 的"工具"，统一编排
- 不自研 LLM；接集团或外部大模型 API（私有化部署根据集团合规要求）
- 不做"自研知识图谱平台"；图谱能力用"图谱 + 向量"两条路径，工具化而非平台化

### 13.8 组织与团队

| 阶段 | 团队规模 | 关键角色 |
|---|---|---|
| P8 | 3-5 人 | 产品 + 后端 + Agent + 前端 + 设计 |
| P9 | 10-15 人 | 加：数据 / AI、IoT 接入、运营 / 品牌对接、合规 |
| P10 | 25-40 人 | 加：客户端（iOS / Android）、风控、数据科学家、保险合作 BD |
| P11-P12 | 50-80 人 | 加：商业化 BD、海外 / 外部、平台化产品、客服 / 增长 |

关键岗位（稀缺、先招）：
- 集团品牌对接 BD（P9 起，每个重点品牌 1 人）
- 数据治理与合规负责人（P9 起，越早越好）
- IoT 平台架构师（P10 起，决定能不能接入手环 / 冰箱 / 音箱）

### 13.9 关键风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 集团品牌接入慢于预期 | B 端 SaaS 收入不达预期 | 用"先免费 POC + 后付费"降低品牌方决策门槛 |
| 数据合规事故 | 平台级风险 | 早期引入数据治理负责人；产品设计 consent first |
| 用户活跃度低于"高频"门槛 | 护城河难建 | 把"今晚吃什么"做到日级刚需，频次驱动产品迭代 |
| C 端商业化与集团品牌利益冲突 | 内部博弈 | C 端永远不全网最低价，定价权交给品牌方 |
| 大模型幻觉导致健康 / 营养错误 | 信任崩塌 | 强证据链 + 营养依据 + 免责声明 + 营养师审核机制 |
| 集团战略调整 / 资源撤回 | 项目停摆 | 模块化设计，部分能力可独立服务，减小单点风险 |

### 13.10 衡量指标

北极星指标：

```text
每周活跃家庭数 × 家庭决策覆盖率
（用户在粮达健康完成"吃什么"决策的比例）
```

阶段 KPI：

| 阶段 | 北极星指标参考值 | 业务指标 |
|---|---|---|
| P8 | 内部演示场景 100% 跑通 | 比赛评分 + 1 个品牌意向接入 |
| P9 | 周活家庭 1 万 | B 端签约 1-2 个品牌；C 端小范围付费转化 ≥ 5% |
| P10 | 周活家庭 50 万 | B 端 5+ 品牌生产级；C 端留存 ≥ 30%；IoT 设备接入 ≥ 3 类 |
| P11 | 周活家庭 300 万 | 订阅用户 ≥ 50 万；B 端年合同 ≥ 3 个；金融合作 ≥ 1 个 |
| P12 | 周活家庭 1000 万 | 平台年 GMV（间接）/ SaaS 收入 / 数据收入 三足鼎立 |

### 13.11 与现有 P0-P7 的关系

P0-P7 是"健康智能营销 Agent"的内核建设期。P8-P12 是"把内核变成集团基础设施"的扩张期。

```text
P0-P7 内核：报告 → 画像 → 记忆 → Agent → 餐单 → 商城 → 证据链
P8-P12 扩张：C 端触点扩张 + B 端 SaaS 扩张 + 数据飞轮扩张 + 集团治理扩张
```

P8 是衔接点：把 P0-P7 的能力打包成"集团领导能听懂的集团级平台故事"，并跑通 1-2 个品牌 PoC。P9 起不再以"比赛"为目的，开始为"上线"而战。
